# WatchGPT

Realtime ChatGPT voice for Apple Watch. **Two-target SwiftUI project**: a watchOS app that captures/plays audio and an iOS companion app that holds the OpenAI Realtime WebSocket. They talk over `WatchConnectivity`. Sideload only — never distribute a build.

## Why two targets

The watchOS networking stack does not let third-party apps open arbitrary outbound TLS WebSockets — empirically tested with `URLSessionWebSocketTask`, `URLSession+waitsForConnectivity`, and `NWConnection+NWProtocolWebSocket`, all of which fail (`-1009`, `cancelled`, or `ENETDOWN [POSIX 50]`) even with the watch on its own cellular and Apple's own apps proven to be passing data. HTTPS data tasks succeed because they're routed through Apple-managed paths a WS task cannot use. The iPhone target sidesteps the entire watchOS networking limit by holding the WebSocket itself and relaying audio to the watch over Bluetooth.

## Layout

- `WatchGPT/` — watchOS target (Swift 5, deploymentTarget 11.0)
  - `WatchGPTApp.swift` — entry point; `AppConfiguration.registerDefaults()`
  - `Views/` — `ContentView`, `SettingsView`, `RealtimeTranscriptBubble`
  - `Services/`
    - `RealtimeVoiceSession` — `WCSession` client; sends `start`/`stop`/audio to phone, receives audio + events back; runs the phase state machine
    - `RealtimeAudioIO` — `AVAudioEngine` mic + `AVAudioPlayerNode`, 24 kHz PCM16
  - `Models/RealtimeTranscriptLine.swift`
  - `Support/AppConfiguration.swift` — minimal: `speakReplies` toggle, `maxStoredMessages`. **No API key on the watch.**
  - `Config/WatchGPT.xcconfig` — committed; intentionally empty (no secrets on the watch)
  - `Info.plist` — mic usage description, `WKApplication`, `WKCompanionAppBundleIdentifier = dev.watchgpt.app`
- `WatchGPTPhone/` — iOS target (Swift 5, deploymentTarget 17.0)
  - `WatchGPTPhoneApp.swift` — entry; activates `PhoneRealtimeBridge` on launch
  - `Views/` — `PhoneContentView` (status), `PhoneSettingsView` (API key)
  - `Services/PhoneRealtimeBridge.swift` — `WCSessionDelegate` ↔ OpenAI `URLSessionWebSocketTask`. Forwards Watch→OpenAI and OpenAI→Watch.
  - `Support/PhoneConfiguration.swift` — API key plumbing (UserDefaults override → bundle default), realtime constants (model `gpt-realtime-2`, voice `marin`, eagerness `medium`, reasoning effort `low`), endpoint URL builder
  - `Config/WatchGPTPhone.xcconfig` — committed; falls through to `LocalSecrets.xcconfig` (gitignored, generated by `npm run configure:phone`)
  - `Info.plist` — `WATCHGPT_OPENAI_API_KEY` from xcconfig, standard iOS keys
- `Shared/RealtimeMessages.swift` — message envelope shared by both targets (compiled into each)
- `scripts/configure-phone.js` — reads root `.env` (or `OPENAI_API_KEY` env var) and writes `WatchGPTPhone/Config/LocalSecrets.xcconfig`
- `project.yml` — XcodeGen spec. `WatchGPTPhone` embeds `WatchGPT`. Run `xcodegen generate` after structural changes.

## Architecture

```
[Watch app] ──WC sendMessage / sendMessageData (Bluetooth)── [iPhone app] ──WSS── [OpenAI Realtime]
```

- Watch captures mic at 24 kHz PCM16, sends each chunk via `WCSession.sendMessageData`.
- iPhone receives audio data, base64-encodes it, sends to OpenAI as `input_audio_buffer.append`.
- iPhone receives OpenAI events. Audio deltas → `sendMessageData(audio)` to watch. Transcript / state events → `sendMessage([type:…])` to watch.
- OpenAI handles `create_response` and `interrupt_response` (both `true`); the watch does not emit `response.create` / `response.cancel`.

## Invariants

- **API key lives only on the iPhone.** Watch has no key plumbing at all.
- **Both apps must be running for a session.** `WCSession.sendMessage` is foreground-to-foreground. If iPhone is suspended, the first watch `start` may fail; iPhone must be open. Future improvement: `BGTaskScheduler` / extended runtime for iPhone background pickup.
- **Message envelope:** `RealtimeMessage.encode(_:payload:)` → `[String: Any]` keyed by `RealtimeMessageKey.type` (`"t"`) and `RealtimeMessageKey.text` (`"x"`). Audio is sent as raw `Data` via `sendMessageData`, not wrapped — channel is dedicated.
- **Audio format:** 24 kHz PCM16 mono in both directions. Don't change without updating both sides.

```sh

# regenerate the Xcode project after adding/removing .swift files or changing project.yml
```

## Conventions / gotchas

- `LocalSecrets.xcconfig` and root `.env` are gitignored. Never commit either.
- The watch's xcconfig is intentionally empty — if you find yourself adding the API key there, you've crossed a trust boundary.
- Both apps need `WCSession.activate()` at launch. We do this in their respective `App.init()` paths via the bridge / session.
- `WCSession.sendMessage` errors don't always surface to the recipient — log/inspect via `errorHandler:` on the sender.
- `sendMessageData` is for binary; `sendMessage([String: Any])` is for control + text. Don't try to wrap audio in a dict — `sendMessageData` is the dedicated channel.
- After adding/removing `.swift` files, run `xcodegen generate`. The pbxproj has explicit refs.
- `RealtimeVoiceSession` and `PhoneRealtimeBridge` both handle old-and-new OpenAI realtime event names (`response.audio.delta` / `response.output_audio.delta`). Don't drop one set without testing.
- This is a personal sideloading project. The build embeds your live API key — never distribute it.
